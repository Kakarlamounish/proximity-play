Routes covered: `/` (Maps.tsx), `/discover` (Discover.tsx), `/live` (Live.tsx), `/ar` (ARView.tsx), `/join/:inviteCode` (JoinBubble.tsx).

Key components/hooks traced: `FriendsMap.tsx`, `Map.tsx`, `useLocation.tsx`, `useGeofencing.ts` (unused/orphaned — see note), `useBatterySaver.ts`, `useSafetyMonitor.ts`, `BubbleFilters.tsx`, `HangoutZonePanel.tsx`, `DeadDropPanel.tsx`, `useDeadDropStore.ts`, `CreateARPinDialog.tsx`, `CreateStoryDialog.tsx`, `CreateEventDialog.tsx`, `PrivacyScheduleDialog.tsx`, `EmergencyShareButton.tsx`, `TripSharingPanel.tsx` + `useTripStore.ts`, `PWALocationBanner.tsx`, `ARScene.tsx`, `ExplorationBadge` (both `components/ExplorationBadge.tsx` and `components/ui/exploration-badge.tsx` — both orphaned/unused), `MobileBottomSheet.tsx`.

## Route: / — Maps
**Auth requirement:** logged-in. `Navigate to="/auth"` if `!user && !loading`. Additionally gated on profile completeness: a `useEffect` queries `profiles.first_name`/`age`; if either is missing it redirects to `/profile-setup` before the map renders (`profileChecked` stays false, showing a spinner in the interim).

### UI inventory
- **Buttons:** "Memory Lane" toggle (adds/removes `?memory=1`, orange highlight when active, haptic on tap), "Filters" toggle (opens a popover, dismisses first-run tooltip), Emergency Share button (delegates to `EmergencyShareButton`), map style picker (Dark/Satellite/Terrain/Street), "Center on me" floating button, Ghost Mode `Switch` (inverted logic: switch ON = sharing OFF), per-friend popup buttons "Chat" / "On my way" / "Halfway" (only shown if their callback prop is supplied — "Halfway" additionally requires `myLocation` to be known).
- **Inputs/forms:** none directly on this page (chat/profile/friends are rendered inside a bottom sheet as full sub-pages).
- **Modals/dialogs/sheets:** `MobileBottomSheet` for `?sheet=messages|profile|friends` (query-param driven), a second `MobileBottomSheet` for "On my way" trip sharing (`TripSharingPanel`), Filters popover, first-run tooltip popover, map-style picker popover.
- **Distinct states:** auth-redirect, profile-incomplete redirect, full-page loading spinner (auth/profile check), map "Loading Snap Map…" spinner (inside `FriendsMap`), map load-error state with "Try again" (network/RLS failure fetching friends), geolocation error banner with "Retry" (permission denied / unavailable / timeout — each with a distinct message), realtime-disconnect banner with exponential-backoff auto-reconnect (1s→30s cap) and "Refresh" button, empty friends bar ("no unread messages" message when `onlyUnread` filter yields nothing but friends exist), populated map with clustered/animated friend markers.
- **Workflows:** (1) Open a friend's popup → tap "Chat" → opens Messages in bottom sheet AND fires a fire-and-forget mark-as-read update for that friend's unread messages. (2) Tap "On my way" → opens Trip sheet pre-filled with friend's location → `TripSharingPanel` geocodes/creates a `trips` row. (3) Tap "Halfway" → computes the arithmetic midpoint of `myLocation` and friend's lat/lng (not a routed midpoint) and opens the same Trip sheet. (4) Toggle Ghost Mode → writes `profiles.ghost_mode` and shows a toast; friends stop seeing you (this write depends on being online — no offline queue).

### Acceptance criteria
1. Given an unauthenticated user visits `/`, when the page loads, then they are redirected to `/auth`.
2. Given an authenticated user with no `first_name`/`age` on their profile, when `/` loads, then they are redirected to `/profile-setup` before ever seeing the map.
3. Given geolocation permission is granted and the user has friends with shared locations, when the map loads, then friend pins render clustered, "You are here" pulsing marker renders at the user's own location, and the friends bar at the bottom lists them sorted by most-recent message then name.
4. Given the user taps "Ghost Mode" on, when the toggle fires, then `profiles.ghost_mode` is set true, a confirmation toast appears, and the user's own marker/visibility to friends is expected to stop (assuming friends re-fetch).
5. Given the user taps "Memory Lane", when toggled on, then a heatmap layer fetches the last 30 days of `location_history` for the current user and renders a gradient legend (blue→red = rare→hot); toggling off removes the layer without re-fetching.
6. Given a friend pin popup is opened, when "Chat" is tapped, then unread messages from that friend are marked `viewed_at` and the Messages overlay opens scoped to that friend.
7. Given the realtime channel drops (CHANNEL_ERROR/TIMED_OUT/CLOSED), when reconnect fails repeatedly, then a "Live updates paused" banner shows and retries with backoff (1s, 2s, 4s… capped at 30s) until `SUBSCRIBED`, at which point friend data is force-refreshed.
8. Given `map:showFriends`/`map:showFriendsBar`/`map:onlyUnread` filter toggles are changed, when the user reloads the page, then their choices persist via `localStorage` per key (not per-user-scoped, so a shared/public device retains the last user's filter prefs).

### Risk-based edge cases
- [High] Geolocation permission denied: `FriendsMap`'s own watchPosition surfaces a specific banner ("Location access denied…") with Retry, but the "You are here" marker and Ghost-mode-dependent writes never happen — Emergency Share button then always fails with "Location unavailable" since `myLocation` stays null.
- [High] Ghost Mode / ghost_mode write happens only when online; if the network request silently fails (not awaited/caught robustly beyond a console.error), the toast still doesn't fire an error state, so a user could believe they are hidden from friends when the write failed.
- [Med] GPS drift/inaccurate coordinates: `enableHighAccuracy` is disabled when `useBatterySaver().saverActive` (low battery or backgrounded tab), and `maximumAge` can be up to 60s — a stale/cached fix can misplace the user's pin or "Halfway" midpoint by a significant distance in fast-moving scenarios (car, transit).
- [Med] No nearby/friend data: zero friends renders an essentially empty map centered at `[20, 0]` (mid-Atlantic) with no explicit "no friends yet" empty-state — could look like a rendering failure to a new user.
- [Med] Realtime reconnect loop: the channel name includes `Date.now()-retryAttempt`, generating a brand-new channel subscription on every retry — under sustained connectivity issues this could churn many stale channels; only cleaned up via the effect's teardown, so a rapid mount/unmount cycle (route flapping) could leak subscriptions momentarily.
- [Med] Battery saver mode interaction: polling interval jumps to 60s (low battery) or 120s (backgrounded tab) — combined with `maximumAge`, a friend's map position can visibly lag by up to 2 minutes with zero UI indication that battery saver is throttling the map (unlike Live.tsx, which explicitly shows a "Battery Saver: GPS polling reduced…" banner — Maps.tsx has no equivalent banner for map users).
- [Low] `localStorage` filter/tooltip state (`map:showFriends`, `map:tips:filters`, `map:tips:popupActions`, friends cache) is not namespaced per-user beyond the friends cache key — filter toggles bleed across accounts on a shared browser profile.
- [Low] Offline/PWA: `PWALocationBanner` explains background-location limits once per install, but there is no offline queue for pending `profiles.latitude/longitude` or `ghost_mode` writes — actions taken fully offline are simply lost (fire-and-forget `.then`/`.catch(console.error)`).
- [Low] Map render failure: if Leaflet/tile CDN requests fail (offline, ad-blocker, CORS), there is no distinguishable "map tiles failed to load" state — only the data-fetch error path (`loadError`) is handled; a pure tile-render failure would show a blank/grey map with markers "floating" with no basemap.

## Route: /discover — Discover
**Auth requirement:** logged-in in practice (all Supabase calls gate on `user`), but the page itself does not redirect unauthenticated users away — an unauthenticated visitor sees the location-permission gate and a page that never populates (no explicit auth redirect exists in this file, unlike Maps/Live/ARView).

### UI inventory
- **Buttons:** "Add Friend" (per nearby user; becomes disabled "Already Friends" / "Request Pending" based on state), "Join" (per bubble), tab switcher People/Bubbles.
- **Inputs/forms:** Search box (client-side filter over already-fetched results, by name/interests for people, by name/description for bubbles), Radius slider (1–50 km, refetches server-side on change), Interest `Select` and Sort `Select` (bubbles tab, via `BubbleFilters`).
- **Modals/dialogs:** none — all inline cards.
- **Distinct states:** location-loading skeletons, location-permission-required error card (distinct from generic error, shown when `!myLocation`), results-loading skeletons (separate from location loading), empty-state cards ("No users found" / "No bubbles found") per tab, populated grid.
- **Workflows:** (1) Adjust radius/interest/sort → triggers a full re-fetch of both nearby users and bubbles (two separate queries, each independently filtered/sorted client-side after fetch). (2) Send friend request → optimistic re-fetch of nearby users to refresh pending-state badges. (3) Join bubble → optimistic re-fetch of nearby bubbles.

### Acceptance criteria
1. Given geolocation is denied or unsupported, when the page loads, then a "Location Access Required" card is shown with the specific browser error message (or a generic fallback), and no nearby-user/bubble queries run.
2. Given location is available, when the radius slider changes, then both people and bubbles re-query and re-render within the new radius, and the label updates to "Radius: N km" live.
3. Given a nearby user is already a friend, when the list renders, then their card shows a disabled "Already Friends" button instead of "Add Friend".
4. Given a nearby user has a pending sent friend request, when the list renders, then their card shows a disabled "Request Pending" button.
5. Given a friend request insert returns a Postgres unique-violation (23505), when this happens, then a "Already sent" toast is shown instead of a generic error.
6. Given a bubble join insert returns a unique-violation (23505), when this happens, then an "Already a member" toast is shown instead of a generic error.
7. Given `sortBy` = "popular"/"members" on the Bubbles tab, when applied, then bubbles are ordered descending by `member_count` (both options currently produce the identical ordering — worth confirming this is intentional, since "popular" and "members" read as different concepts to a user).
8. Given the search box has text, when typed, then filtering is purely client-side over the already-radius-filtered result set (not a fresh server query) — typing a name outside the current radius/interest filter will never surface it.

### Risk-based edge cases
- [High] No auth redirect: unlike every other page in this slice, `Discover.tsx` has no `<Navigate to="/auth">` guard. An unauthenticated user hitting `/discover` directly will see the location-permission-required (or loading) card indefinitely since every fetch function early-returns on `!user`, with no messaging that login is required.
- [High] Distance math correctness: `calculateDistance` uses a standard Haversine formula in km and looks correct; however both people and bubbles distance-filter with a strict `<= radiusKm[0]` — a nearby user/bubble sitting exactly at the boundary is a coin-flip on floating-point rounding, and there is no server-side (PostGIS/RPC) filtering — the entire `profiles` table (all rows with non-null lat/lng) is pulled to the client and filtered in JS, which will not scale and also leaks every user's raw coordinates to any authenticated client regardless of distance/privacy settings (only `ghost_mode`/ RLS on `profiles` would prevent this, and this query does not appear to check `ghost_mode` at all).
- [High] Privacy: the nearby-users query (`profiles.select('id, first_name, ..., latitude, longitude').neq('id', user.id)`) does not filter out users who have Ghost Mode enabled elsewhere in the app (Maps/Live use `ghost_mode` to hide a user's pin) — Discover's "nearby people" list may surface users who believe they are hidden.
- [Med] GPS drift/inaccurate coordinates: `useLocation` requests a single `getCurrentPosition` fix with `maximumAge: 300000` (5 min) and never watches position — the radius/distance calculations use a potentially stale fix; there's no "Refresh my location" affordance on this page (only the initial fetch on mount/user-change).
- [Med] No nearby data: radius set to the 1km minimum in a sparse area yields "No users found"/"No bubbles found" — message suggests "adjusting your search radius", but the slider's current value isn't highlighted/scrolled-to, so discoverability of the fix is weak.
- [Med] Expired/private bubbles: the bubbles query only filters `is_private = false`; there is no expiry concept on bubbles here (n/a), but a bubble with `member_count` stale relative to actual membership (no live recompute) could show inflated/incorrect member counts if that column isn't kept in sync by triggers.
- [Low] Offline/PWA: no offline handling — a request failure while offline surfaces a generic "Failed to load nearby users" toast with no retry button (unlike Maps' `FriendsMap`, which has an explicit "Try again").
- [Low] Battery saver: N/A directly (no continuous watch), but the single fixed-location fetch means Discover never benefits from or is throttled by `useBatterySaver` — results can go stale if the user travels after opening the page and never triggers a refetch.

## Route: /live — Live
**Auth requirement:** logged-in. `Navigate to="/auth"` if `!user && !loading`. No explicit profile-completeness check (unlike Maps).

### UI inventory
- **Buttons:** bubble selector chips (switch `selectedBubble`), "Post a Story" / "Create Event" / "Drop AR Pin" (open respective dialogs), Ghost Mode-adjacent controls: "Enable/Disable Ghost Mode", "Show/Disable Heatmap", "Pause/Enable Location Sharing" (labelled `privacyEnabled`, inverted semantics — button reads "Enable Location Sharing" while sharing is currently paused), Emergency Share button, "Privacy Schedule" button (opens `PrivacyScheduleDialog`), disabled placeholder status buttons ("☕ Coffee" etc. under Status tab — non-functional, `disabled` hardcoded), tab bar (Map/Status/Activity/Drops/Trip/Zone).
- **Inputs/forms:** Proximity Radius numeric input (10–1000m, drives the in-page proximity alert only, not persisted), inline bubble-chat text input + Send button (component-local state only — not backed by the `bubble-chat-*` Supabase broadcast channel it subscribes to receiving from, so **a user's own sent messages are added to local `chatLog` directly and never actually broadcast to other members** — this chat is effectively non-functional for real 2-way chat), `PrivacyScheduleDialog` start/end time inputs.
- **Modals/dialogs:** `CreateStoryDialog`, `CreateEventDialog`, `CreateARPinDialog`, `PrivacyScheduleDialog` — all gated on having a resolved `userLocation` (see below), plus embedded panels-as-tabs: `DeadDropPanel`, `TripSharingPanel`, `HangoutZonePanel`.
- **Distinct states:** full-page loading spinner, "No bubbles joined" empty state with "Find Bubbles" button (routes to `/`), per-bubble tabs each with their own loading/empty states (badges list, activity feed "No recent activity yet", drops/zones empty states), Battery Saver banner (visible, unlike Maps) when `saverActive`, geofence-entry alert banner (blue, "You entered the bubble zone!"), proximity alert banner (red, "User nearby!") with haptic fired only on alert's rising edge.
- **Workflows:** (1) Select a bubble → publishes location to `live_locations` (5 min TTL), computes geofence distance to bubble center, and — as a side effect of the *same* location-publish callback — silently checks/awards an "Explorer" badge if the user is >2 km from the bubble center. (2) Adjust Proximity Radius → live re-evaluates `proximityAlert` against currently-known `liveLocations`. (3) Post Story/Create Event/Drop AR Pin — each dialog independently derives `userLocation` from `liveLocations.find(l => l.user_id === user.id)`, i.e. from the **already-published-to-DB** row, not directly from the GPS watch.

### Acceptance criteria
1. Given the user has joined zero bubbles, when `/live` loads, then a "No bubbles joined" card is shown with a "Find Bubbles" button linking to `/`.
2. Given a bubble is selected and geolocation succeeds, when a position update fires, then a row is upserted into `live_locations` with a 5-minute `expires_at`, and if the distance to the bubble center is < 200m a geofence banner appears; if it later exceeds 200m the banner clears.
3. Given the user is > 2000m from the bubble center and has not yet earned the "Explorer" badge, when a location update fires, then the badge is inserted into `user_badges` exactly once (guarded by an `alreadyEarned` check against the current in-memory `userBadges` fetch, not a DB unique constraint).
4. Given another bubble member's `live_locations` row is within `proximityRadius` meters, when this is first detected, then a haptic (`friendNearby` pattern) fires exactly once (only on the transition from no-alert to alert, tracked via `prevProximityAlert` ref) and a red "User nearby!" banner shows.
5. Given `batterySaver.saverActive` is true (low battery or backgrounded tab), when this page is active, then a visible amber "Battery Saver: GPS polling reduced to every Ns" banner is shown and `enableHighAccuracy` is disabled for the location watch.
6. Given the user opens "Post a Story"/"Create Event"/"Drop AR Pin" before their own `live_locations` row has round-tripped through Supabase, then the dialog's submit button is disabled (`!userLocation`) even though the device's raw GPS coordinates are already known elsewhere on the page — there is no loading indicator explaining why submit is disabled.
7. Given the user types in the "Bubble Chat (demo)" box and hits Send, then the message is appended to local `chatLog` state only — it is never published to the `bubble-chat-{bubbleId}` broadcast channel the page subscribes to, so no other bubble member ever receives it.

### Risk-based edge cases
- [High] Bubble chat is decorative only: the send handler never calls `chatChannel.send(...)` / `supabase.channel(...).send(...)`, it just does `setChatLog(log => [...log, ...])` locally. Any QA pass that assumes "Bubble Chat" is a working real-time chat feature will find it silently broken — messages never leave the sender's device.
- [High] Location spoofing / trust: all location-derived side effects (Explorer badge award, geofence "entered bubble zone" alert, proximity "user nearby" alert, dead drop/hangout-zone unlocks) trust the browser-reported `navigator.geolocation` coordinates with no server-side plausibility check (e.g., max speed between two consecutive fixes). A user can spoof GPS (devtools override, mock-location apps) to farm the Explorer badge or fake "arriving" at a hangout zone/dead drop without traveling.
- [High] Unthrottled location writes: unlike `FriendsMap.tsx` (which gates writes with an explicit `writeIntervalMs`/`lastWriteAt` check), this page's `publishLocation` upserts to `live_locations` **and** queries `badges`/`user_badges` on every single `watchPosition` callback with no debounce — with `enableHighAccuracy` and a fast-changing GPS fix (e.g., in a car), this can fire many DB writes/reads per minute per active user.
- [Med] `PrivacyScheduleDialog` and `EmergencyShareButton` both render `<DialogHeader>`/`<DialogTitle>` as a **sibling of, not a child inside,** `<DialogContent>` (e.g. `PrivacyScheduleDialog.tsx` lines 43–46, `EmergencyShareButton.tsx` lines 45–48). Because Radix's `Dialog.Root` renders its children directly (only `DialogContent`'s internal `Presence`/portal is gated on `open`), the header/title text ("Privacy Scheduling", "Share Live Location (Emergency)") is likely rendered unconditionally in the page flow regardless of the dialog's open state — verify visually; if confirmed, stray heading text will appear inline on the Live page at all times, not just when the dialog is opened.
- [Med] Expired ephemeral content: `live_locations` rows expire after 5 minutes; if a tab is backgrounded (mobile OS suspends timers/geolocation) longer than that, the user disappears from bubble-mates' view with no "you're offline" self-indicator on this device.
- [Med] Battery saver mode: `enableHighAccuracy: !batterySaver.saverActive` combined with the unthrottled write path above means low-accuracy fixes are written to the DB (and used for geofence/proximity math) just as eagerly as high-accuracy ones — proximity alerts and geofence entry can misfire near boundaries under battery saver.
- [Med] No nearby data / empty states: "Activity" tab shows "No recent activity yet" correctly, but the Map tab's "Live location sharing requires database updates that will be available after the next deployment" copy is stale placeholder text left in production code — misleading to QA and end users about feature completeness.
- [Low] Concurrent badge award race: two rapid location updates before the first `user_badges` insert completes could both pass the `alreadyEarned` check (computed from a stale local fetch) and attempt duplicate inserts — only safe if `user_badges` has a DB-level unique constraint on `(user_id, badge_id)`, which is not verifiable from this file.
- [Low] Offline/PWA: no offline queue for `live_locations` upserts, badge awards, or story/pin/event creation — anything attempted offline fails silently (console-logged) with a generic toast at best.

## Route: /ar — ARView
**Auth requirement:** logged-in. `Navigate to="/auth"` if `!user && !loading`.

### UI inventory
- **Buttons:** back chevron (`navigate(-1)`), "Enable Camera" (primary CTA, triggers device-orientation permission request *then* `getUserMedia`), "Stop" (stops camera + tracks), "Resume" (re-shows camera if previously granted).
- **Inputs/forms:** none.
- **Modals/dialogs:** none — single full-screen camera view with overlay HUD.
- **Distinct states:** camera-not-started placeholder ("Point camera at the world" + Enable Camera button), camera-error state (permission denied / device unsupported — shows `err.message`), camera-active with AR overlay, "Finding friends…" vs "No friends sharing location nearby" pill (distinguished by `loadingFriends`), GPS-not-yet-available ("Waiting for GPS...") vs GPS-available (`lat, lng` readout, 4-decimal precision) footer state, live compass ring (rotates with `deviceHeading`).
- **Workflows:** (1) Tap Enable Camera → `requestDeviceOrientationPermission()` (iOS 13+ gesture-gated prompt) → `getUserMedia({video: {facingMode: 'environment'}})` → on success, camera feed + `ARScene` (Three.js 3D-projected friend sprites reacting to alpha/beta/gamma) render; on failure, `cameraError` is set and shown. (2) Friends refresh every 15s via `get_friend_locations` RPC; distance/bearing recomputed via Haversine + bearing formulas whenever `myLocation` or the friends list changes. (3) Stop/Resume toggles the media stream without re-requesting permissions (tracked via `permissionGranted`).

### Acceptance criteria
1. Given camera permission has never been granted, when the page loads, then the "Point camera at the world" placeholder is shown with an "Enable Camera" button — no camera/mic access is requested until the button is tapped (user-gesture requirement respected, important for iOS).
2. Given the user taps "Enable Camera" and grants both device-orientation and camera permissions, then the live video feed renders full-bleed, the crosshair and compass ring overlay appear, and `ARScene` begins rendering friend sprites.
3. Given camera permission is denied, when `getUserMedia` rejects, then `cameraError` shows the browser's error message plus static guidance text ("Enable camera access in your browser settings…"), and no "Resume" button is available (only re-shown once `permissionGranted` has been true at least once in this session).
4. Given a friend has a non-null lat/lng from `get_friend_locations` and is not the current user, when `myLocation` is known, then their marker's distance (Haversine, meters) and bearing (0–360° from true north) are computed and passed to `ARScene` for 3D projection.
5. Given zero friends are sharing location, when the camera is active, then a pill reads "No friends sharing location nearby" (once the initial fetch completes) or "Finding friends…" (while `loadingFriends` is true).
6. Given the device orientation changes (compass heading), when `deviceorientation` fires, then the compass ring rotates opposite to `deviceHeading` and the "N" label counter-rotates to stay upright, and `ARScene`'s Three.js camera quaternion updates using the alpha/beta/gamma + screen-orientation correction so friend sprites stay anchored to their real-world bearing as the phone tilts/rotates.

### Risk-based edge cases
- [High] Geolocation permission denied/unavailable: the GPS `watchPosition` call (lines 124–129) has **no error callback at all** — if location is denied, `myLocation` simply never becomes non-null, the footer shows "Waiting for GPS..." forever, and there is no way for the user to discover *why* (no distinction from "still acquiring a fix"). This is a materially worse experience than every other page in this slice, all of which surface a specific permission-denied message.
- [High] Device-orientation permission denied (iOS): `requestDeviceOrientationPermission()`'s boolean result is never checked in `startCamera` — if the user denies the motion/orientation prompt, the camera still starts and friend sprites will render at a fixed/incorrect position (heading frozen at whatever `deviceHeading` last was, defaulting to 0/north) with no error shown to the user; this looks like a rendering bug rather than a permission issue.
- [Med] GPS drift/inaccurate coordinates: no `enableHighAccuracy`/`maximumAge`/`timeout` options are passed to `watchPosition` here (defaults apply), so on some devices this may return a low-accuracy, cell-tower-level fix — bearing/distance to friends could be off by tens to hundreds of meters, causing AR arrows to point the wrong direction.
- [Med] No `useBatterySaver` integration on this page (unlike Live.tsx/FriendsMap.tsx) — continuous camera + WebGL rendering + `enableHighAccuracy`-default GPS watch + Three.js `requestAnimationFrame` loop is one of the most battery-intensive screens in the app with zero low-power mitigation.
- [Med] Stale/very-far friends: there is no maximum-distance cutoff — a friend thousands of km away still renders a sprite (using the log-compressed `projectedDistance`), which is visually meaningless in an AR "point your phone" context and could confuse users into thinking a very distant friend is nearby.
- [Med] Camera device unavailable (desktop/no rear camera, or camera in use by another app): `getUserMedia` rejects and `err.message` (browser-dependent, sometimes generic like "Could not start video source") is shown verbatim — no friendlier fallback copy.
- [Low] Backgrounding/tab-switch: no visibility-change handling to pause the camera stream or GPS watch when the tab is hidden — camera light/battery drain continues if the OS doesn't auto-suspend it.
- [Low] Offline/PWA: the 15-second friend-location refresh (`get_friend_locations` RPC) will simply fail silently (`console.error`) when offline, leaving stale bearings/distances with no "offline" indicator.
- [Low] Screen/device orientation changes (portrait↔landscape) update `screenOrientation` via the `orientationchange` listener and resize the renderer via `resize`, but there's no explicit test coverage implied for orientation-change mid-AR-session; worth a manual check that sprites don't jump/misalign during the rotation animation.

## Route: /join/:inviteCode — JoinBubble
**Auth requirement:** public (page renders for anonymous visitors — invite details are fetched without requiring `user`); joining the bubble itself requires auth (button becomes "Sign in to Join" and redirects to `/auth?returnTo=/join/:inviteCode` otherwise).

### UI inventory
- **Buttons:** "Join Bubble" / "Sign in to Join" (same button, label + icon swap based on auth state), "Go to Bubble Chat" (shown after joining or if already a member — navigates to `/messages` with `state.selectedBubbleId`), "Go to Home" (ghost button, always present), "Go to Home" (also shown standalone on the invalid-invite error state).
- **Inputs/forms:** none — purely a landing/confirmation page driven by the `:inviteCode` URL param.
- **Modals/dialogs:** none.
- **Distinct states:** loading spinner (fetching invite + auth resolving), invalid-invite error card (covers: missing code, not found/inactive, expired, max-uses reached, bubble no longer exists, generic fetch failure — all collapse to the same visual "Invalid Invite" card, differentiated only by the error text), "already a member" state, "welcome/joined" success state (green checkmark), default "you're invited" state with the join CTA.
- **Workflows:** (1) Load page → fetch `bubble_invites` by code (`is_active=true`) → validate `expires_at`/`max_uses` client-side → fetch the `bubbles` row → if authed, check `bubble_memberships` for existing membership. (2) Tap Join (authed) → insert into `bubble_memberships` → on success, **non-atomically** read-then-write-increment `bubble_invites.uses` → insert a `notifications` row for the bubble creator (skipped if the joiner *is* the creator) → show success state. (3) Tap "Sign in to Join" (unauthed) → `navigate('/auth?returnTo=/join/:inviteCode')`.

### Acceptance criteria
1. Given an invite code that doesn't exist or whose `is_active` flag is false, when the page loads, then an "Invalid Invite" card shows "This invite link is invalid or has expired."
2. Given an invite whose `expires_at` is in the past, when the page loads, then the card shows "This invite link has expired." (checked client-side after the initial fetch succeeds, distinct message from case 1).
3. Given an invite whose `uses >= max_uses`, when the page loads, then the card shows "This invite link has reached its maximum usage limit."
4. Given a valid invite but the referenced `bubbles` row is missing (deleted bubble), when the page loads, then the card shows "The bubble associated with this invite no longer exists."
5. Given an authenticated user who is already a member of the target bubble, when the page loads, then the "You're already a member" state is shown directly, bypassing the join CTA.
6. Given an unauthenticated visitor, when the page loads with a valid, non-expired, non-maxed invite, then the bubble preview renders (name, interest tag, description, member count) and the CTA reads "Sign in to Join".
7. Given an unauthenticated visitor taps "Sign in to Join", then they are routed to `/auth?returnTo=/join/:inviteCode`.
8. Given a join insert returns a unique-violation (23505 — already a member, e.g. from a race), then the UI transitions to the "already a member" state rather than showing a generic error.
9. Given a successful join, then a `notifications` row is created for the bubble's `creator_id` (unless the joiner is the creator themselves), and the invite's `uses` counter is incremented.

### Risk-based edge cases
- [High] **`returnTo` is never honored**: `Auth.tsx` has `if (user && !loading) return <Navigate to="/" replace />;` — a hardcoded redirect to `/`, and never reads any `returnTo`/`?returnTo=` query parameter. A user who lands on `/join/:code`, isn't logged in, taps "Sign in to Join", authenticates successfully, is dropped on the Maps page (`/`) with **no memory of the invite code** — they must re-find/re-click the original invite link to actually join. This breaks the single most important conversion path on this page.
- [High] Concurrent joins / invite-uses race: the "increment uses" logic (lines 138–156) is a **non-atomic read-then-write**: `select('uses')` then `update({uses: currentInvite.uses + 1})`. Two users joining via the same invite link within the same read-write window can both read the same `uses` value and both write `uses+1`, undercounting real usage — an invite with `max_uses` could be used more times than intended (the membership insert itself is protected by a unique constraint, but the *usage counter* enforcement is not).
- [Med] Dead/no-op code smell directly adjacent to the race above: line 141 — `.update({ uses: supabase.rpc ? undefined : 1 })` — `supabase.rpc` is always a truthy function reference, so this always evaluates to `.update({ uses: undefined })`, an effectively no-op call executed immediately before the real read-then-increment update. Confirm this isn't masking an intended (and safer) atomic RPC-based increment that was never wired up.
- [Med] Invite-code invalid/expired/already-member are visually indistinguishable in structure (same red "Invalid Invite" card) except for the message string — a screenshot-based or non-text-reading QA pass could conflate "expired" with "not found" with "wrong code typed". Verify each message string individually rather than just "an error card shows".
- [Med] `bubble.interest_tag[0].toUpperCase()` (line 243) will throw if `interest_tag` is ever an empty string (not null — the earlier `?.` guards only cover null/undefined-shaped errors upstream, not empty-string tags) — worth a data-integrity check on whether `interest_tag` can be empty for any bubble.
- [Low] No geolocation/proximity gating at all on this page — unlike other join flows in social apps, there is no check that the joining user is anywhere near the bubble's location; a user can join a hyper-local bubble from anywhere in the world purely via the invite link.
- [Low] Offline/PWA: no offline handling — if the initial invite/bubble fetch fails due to no connectivity, the generic "Failed to load invite details" message is shown with no retry button and no distinction from a genuinely invalid code.
- [Low] Notification-insert failure (e.g., RLS or missing `notifications` table) is not caught/guarded separately from the join success path — if the `notifications` insert throws, it would propagate into the same `catch` block as the join itself, potentially showing "Failed to join bubble" to a user whose membership *was* actually created successfully (misleading error after a successful join).

## Cross-cutting notes (apply to more than one route above)

- **Dead/orphaned code surfaced during this audit:** `src/hooks/useGeofencing.ts` (a fully-implemented enter/exit geofence-alert hook against `useGeofenceStore`/`useAppStore`) is never imported by any page in the app — the actual geofencing behavior users experience today is the simpler, ad-hoc 200m "bubble zone" check inline in `Live.tsx`. Similarly, both `src/components/ExplorationBadge.tsx` and `src/components/ui/exploration-badge.tsx` are two near-duplicate components, neither of which is ever rendered anywhere (the only import, in `Map.tsx`, is unused dead code). QA should not expect to find a working "geofence enter/exit notification" or "% explored" badge feature in the current build despite the supporting code existing.
- **`src/components/Map.tsx`** (used inside Live.tsx's "Map" tab) is in a rough, half-finished state: it contains a weather overlay wired to a literal placeholder string `'<YOUR_OPENWEATHERMAP_API_KEY>'` (the weather toggle button is fully clickable and will fire a failing fetch to OpenWeatherMap with an invalid key every time), an in-memory-only "Create Event/Bubble" dialog that pushes to the local `bubbles` prop array without ever calling Supabase (changes vanish on refresh/re-render), duplicate/triplicated copies of the same geofence-enter/exit `useEffect`, and several console-only debug handlers (`MapEvents` logs clicks/moves/zooms to console). Anyone testing "Live → Map tab" should expect visible but non-functional controls (weather toggle, in-map bubble creation, route planning between two hardcoded Hyderabad coordinates) — flag these as known-incomplete rather than newly-discovered bugs.
- **Location trust/spoofing** is a recurring theme: none of the five routes validate that reported GPS coordinates are physically plausible (no speed/teleport sanity check server-side). This affects Explorer badge farming (Live), dead-drop/hangout-zone unlocking (Live), geofence "entered bubble zone" alerts (Live), and AR friend bearing accuracy (ARView) — likely worth a single shared risk item in the master risk register rather than repeating per-page in test execution.
- **Duplicate `DialogHeader`-outside-`DialogContent` pattern** appears in at least two components reachable from these routes (`EmergencyShareButton.tsx`, rendered on both Maps and Live; `PrivacyScheduleDialog.tsx`, rendered on Live) — worth a single UI smoke test across both surfaces rather than treating as two unrelated bugs.
