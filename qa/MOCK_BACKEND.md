# Mock Backend (QA / local testing)

This app can run entirely against a **client-side-only mock backend** —
no real network calls, no Supabase project required, no risk to the live
hosted database. It's meant purely for local QA/dev exercising of the app.

The real Supabase client (`src/integrations/supabase/client.ts`) and the
live hosted project are **never touched**. The mock is swapped in only via a
Vite resolve alias, gated behind an env flag, so a normal build/dev run is
completely unaffected.

## How to enable it

Either of these works:

```bash
# One-off, current shell only:
VITE_USE_MOCK_BACKEND=true npm run dev
```

```bash
# Persistent: add to .env.local (create the file if it doesn't exist)
echo "VITE_USE_MOCK_BACKEND=true" >> .env.local
npm run dev
```

It also works for production-mode builds/previews:

```bash
VITE_USE_MOCK_BACKEND=true npm run build
npm run preview
```

When the flag is **unset or not exactly `"true"`**, nothing changes —
`@/integrations/supabase/client` resolves to the real generated client, same
as today.

You'll know it's active when the browser console prints, on load:

```
[mock-backend] Enabled — no real network calls will be made.
Test account: test@example.test / TestPass123!
```

## Seeded test account

| Field | Value |
|---|---|
| Email | `test@example.test` |
| Password | `TestPass123!` |

This account has a fully-populated profile (bio, interests, photo), ~18
accepted friends plus a few pending/declined requests (to test every friend
UI state), membership in 6 bubbles (2 of which it created), unread
notifications, DM and bubble chat history, call logs, a snap score, and a
couple of earned badges — so signing in drops you straight into a populated
app rather than an empty new-user state.

Sign in with email/password at `/auth` — Google OAuth is **not** supported
in the mock (see limitations below), so use the email/password form.

## What works fully

- Auth: email/password sign up & sign in, sign out, session persistence for
  the life of the tab, `onAuthStateChange` firing on sign-in/out.
- All 37+ tables' CRUD via the same `.from(table).select/insert/update/upsert/delete()`
  chain the app already uses, including `.eq/.neq/.gt/.gte/.lt/.lte/.in/.or/.contains/.order/.limit/.range/.single/.maybeSingle`.
- Supabase's embedded-resource select syntax (`.select('*, badge:badges(*)')`,
  `.select('*, creator:profiles!bubbles_creator_id_fkey(first_name)')`) —
  the join column is read from a `!fk_constraint` hint when present, or
  inferred by singularizing the alias/table name and appending `_id`
  otherwise. Added after live QA testing found the initial implementation
  silently dropped every embedded join (see `qa/BUGS.md`'s "Mock-backend
  infra fix" entry).
- Row-level-security equivalent scoping (see "Scoping decisions" below) —
  you can't read/write rows the real RLS policies would also block.
- The two RPCs: `get_friend_locations()` and `get_nearby_dead_drops()`
  (haversine-distance filtered, respects each drop's own radius and
  expiry).
- Storage: upload/getPublicUrl/createSignedUrl/list across all 3 buckets
  (`profile-photos`, `stories`, `voice-notes`), backed by
  `URL.createObjectURL` — uploaded images/voice notes really render.
- Realtime within a single tab: `postgres_changes`, `broadcast`, and
  `presence` (`track`/`untrack`/`presenceState`/`sync`) all work — e.g.
  editing your profile in one component updates a `FriendsMap` subscribed to
  `profiles` changes in the same tab.
- Edge functions: `redeem-referral` actually looks up seeded profiles'
  `referral_code` and creates a `referrals` row.

## Known limitations

- **Google OAuth sign-in is not supported.** `signInWithOAuth` can't fake a
  real redirect round-trip; it resolves with an error and logs a console
  warning. Use email/password instead.
- **WebAuthn/biometric enrollment doesn't really work.** The 4
  `webauthn-*` edge functions return plausible-shaped fake challenge/verify
  responses so the Biometric UI renders without crashing, but the actual
  browser WebAuthn ceremony (`startRegistration`/`startAuthentication` from
  `@simplewebauthn/browser`) still talks to the real platform authenticator
  API — it may succeed or fail depending on your device/browser, and even
  when it succeeds, the "verification" is a rubber stamp, not real
  cryptographic verification.
- **Realtime is single-tab only.** There's no WebSocket and no cross-tab or
  cross-browser delivery. Two windows each running the mock backend are two
  independent worlds — manual "two testers call each other" flows won't see
  each other's realtime events.
- **WebRTC media between two real peers doesn't work either**, for the same
  reason: call signaling (`broadcast`) only reaches other channel instances
  *within the same tab*, and there's normally only one authenticated
  "session" active at a time in the mock, so there's no second peer to
  connect to. The call UI, ringing flow, and `call_logs` state machine all
  work; actual two-way audio/video does not.
- **No data persistence, but the session now survives a reload.** All table
  data lives in memory and a page reload re-runs `seedMockBackend()` from
  scratch (same deterministic dataset, so the seeded test account is
  unaffected) — but *who's signed in* is now persisted to
  `localStorage['mock-backend:session']` (mirroring real supabase-js) so
  that typing a URL directly, hitting refresh, or opening the app in a way
  that causes a full navigation doesn't look like a logged-out session. This
  was added after live QA testing found that every direct-URL navigation was
  being misread as "not authenticated," which isn't how the real (real-Supabase-backed)
  app behaves. **Caveat:** this only cleanly restores the seeded test
  account — a custom account created via sign-up during a session will not
  survive a hard reload, since nothing re-creates that account's row when
  `seedMockBackend()` reseeds only the fixed dataset.
- **Email confirmation is skipped.** Sign-up immediately establishes a
  session (a deliberate deviation from the real project, so a QA tester can
  create an account and start using the app in one step, without an email
  step that has nowhere to deliver to).

## Scoping decisions made where the spec was ambiguous

A few tables' RLS-equivalent rules required judgment calls (documented in
detail as comments in `src/integrations/supabase/mock/rls.ts`):

- **`friendships`**: direct client-side inserts are denied — the app never
  inserts a friendship directly, only via accepting a `friend_requests` row,
  which the mock replicates as a trigger (accepting creates the friendship
  row automatically).
- **`notifications`**: insert is allowed for any authenticated user (not
  just `user_id === self`), because notifications are routinely created by
  other users' actions (e.g. your accepted friend request notifies you).
- **`dead_drops`**: update is allowed for any authenticated user, because
  `markAsViewed()` is called by whoever *finds* the drop, not its creator.
- **`bubble_invites`**: update (the `uses` counter) is allowed for any
  authenticated user for the same reason — the joining user increments it,
  not the invite's creator.
- **`voice_messages`**: read is public and update is any-authenticated-user,
  because `chat_id` may represent either a DM pair or a bubble, and the
  *recipient* (not just the sender) needs to mark a note played — there's
  no clean single owner column to key off, so this falls back to the task's
  own "public read / own write" default guidance.
- **`emergency_shares`**: read is owner-only. The feature name implies
  "trusted contacts" should also be able to read a share, but there's no
  trusted-contacts relationship anywhere in the client code to derive that
  scoping from, so it defaults to the safe (owner-only) fallback.
- **`get_friend_locations()`**: ghost-mode friends are omitted from the
  result entirely (rather than included with nulled-out coordinates) — this
  matches the intent of "ghost mode" as full invisibility, including to
  friends.

For every table not explicitly covered above or in the original scoping
brief, the mock falls back to: SELECT allowed for everyone, INSERT/UPDATE/DELETE
restricted to rows where an obvious owner-ish column (`user_id`,
`created_by`, `creator_id`, `owner_id`, `sender_id`, `reporter_id`,
`blocker_id` — whichever is present) equals the current user.
