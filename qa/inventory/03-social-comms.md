# Social & Communications Feature Inventory — Friends, Messages, Calls, Missed Calls, Camera, Stories

## Route: /friends — Friends
**Auth requirement:** No route-level guard exists in `App.tsx` (no `ProtectedRoute` wrapper) and `Friends.tsx` itself never checks `user` and redirects — unlike Messages/Calls/MissedCalls/Camera. A logged-out visitor can load `/friends`; every data fetch (`fetchProfile`, `fetchFriends`, `fetchSentRequests`, `fetchSuggestedFriends`) early-returns on `!user`, so the page renders with an empty "My Friends" list and a working search box (search itself doesn't require `user`, only excludes `user?.id` which is `undefined`). Effectively public-but-useless rather than truly gated.

### UI inventory
- Buttons: global search input (debounced 300ms), search-range `Select` (1/5/10/25/50/100/500/1000 km / Global), per-result "Add Friend"/"Request Sent"/"Already Friends" (search results + suggested), per-friend "Message", "Audio call" (Phone icon), "Video call" (Video icon), "Remove friend" (UserMinus), "Discover People" (empty state), Friend Requests card Accept (check) / Reject (X) (rendered by `FriendRequests.tsx`).
- Inputs/forms: name search `Input` (`ilike first_name`), range `Select`.
- Modals/dialogs: none on this page (no confirm dialog for "Remove friend" — it fires immediately on click).
- Distinct states: initial loading spinner (friends list), search loading spinner, empty search results, empty "No friends yet", populated friends grid, suggested-friends section (hidden when empty), pending-requests panel (loading/empty/populated).
- Workflows: (1) search by name or browse "Nearby People" by radius → "Add Friend" → row inserted into `friend_requests` (status `pending`) + a `notifications` row → button flips to "Request Sent". (2) Receiver sees it in `FriendRequests` panel (realtime-subscribed) or via the global `FriendRequestNotifier` toast → Accept writes `status='accepted'` + notifies sender → friendship exists via `friendships` table (queried by `Friends`/`Messages`/`Calls` as the union of `user_id_1`/`user_id_2`) → "Message" button deep-links to `/messages` with `state.selectedFriendId`; Phone/Video buttons call `startCall(friend.id, 'audio'|'video', false)` from `CallContext`.

### Acceptance criteria
1. Given a logged-in user with 0 friends, when `/friends` loads, then "No friends yet" empty state is shown with a "Discover People" button routing to `/discover`.
2. Given a search query is typed, when 300ms pass with no further typing, then `searchUsers` runs exactly once with the current query and range.
3. Given `searchRange` is not `'global'` and geolocation succeeds, when no query is typed, then a "Nearby People" list is shown filtered by Haversine distance ≤ selected km.
4. Given a user clicks "Add Friend" on a profile with no existing relationship, when the insert succeeds, then a `friend_requests` row (`status: pending`) and a `notifications` row are created, a success toast appears, and the button becomes disabled and reads "Request Sent" without a page reload.
5. Given a pending request already exists in either direction, when "Add Friend" is clicked, then a destructive toast "Request already sent" is shown and no duplicate row is inserted (best-effort check, see risks).
6. Given two users are already friends, when either searches for the other, then the button reads "Already Friends" and is disabled.
7. Given a friend request is accepted via `FriendRequests`, when the mutation succeeds, then the requester is notified (`friend_request_accepted`), the request disappears from the pending list, and both users' `friendships` queries return each other.
8. Given "Remove friend" is clicked, when the delete succeeds, then the friend disappears from "My Friends" and reappears as a possible "suggested friend" (bubble co-membership) if applicable — with no confirmation step in between.
9. Given "Message"/audio/video buttons are clicked on a friend card, then the app navigates to `/messages` (with the friend pre-selected) or opens the global call UI via `CallContext.startCall`, respectively.
10. Given `suggestedFriends` is computed, then it excludes current friends and the logged-in user, is capped at 6, and is derived only from shared bubble memberships (not from search).

### Risk-based edge cases
- [High] Double-click on "Add Friend": there is no loading/disabled state while the insert is in flight, so two rapid clicks can both pass the "existing request" check before either insert lands, creating two `friend_requests` rows for the same pair.
- [High] `sendFriendRequest`'s duplicate check uses `.single()` on an `OR` filter across both directions; once a request has been sent, rejected, and re-sent (a realistic "spam" pattern), more than one row can match and `.single()` silently returns `null` data (no thrown error surfaced), bypassing the "already sent"/"already friends" guard entirely and allowing unlimited duplicate pending requests to accumulate.
- [High] No blocking/mute feature exists anywhere in the codebase (`grep` for "block" across `src` finds no user-blocking model) — a harassing user cannot be blocked from sending further friend requests, messages, or calls; only "Remove friend" (which doesn't prevent a fresh request afterward) exists.
- [Med] "Remove friend" has no confirm dialog — an accidental click permanently deletes the friendship row with only a toast as feedback, no undo.
- [Med] Client-side Haversine distance filtering fetches up to 100 profiles per range and filters in-browser; for a large user base this both scales poorly and can leak coarse-grained location data of far-away users to the client before filtering.
- [Med] `getMutualFriendsCountBatch` (used for both search results and suggested friends) issues one `friendships` query per candidate user sequentially — with many search results this is an N+1 query pattern that can visibly slow the results list.
- [Low] Logged-out access: since there's no page-level redirect, an unauthenticated visitor sees a fully-rendered but empty/non-functional Friends page instead of being redirected to `/auth`, which is inconsistent with Messages/Calls.
- [Low] Selecting "Global" range while a query is also present ignores the range filter entirely — no visual indication that "Global" overrides km filtering.

## Route: /messages — Messages
**Auth requirement:** logged-in. `Messages.tsx` explicitly does `if (!user && !loading) return <Navigate to="/auth" replace />;` and shows a full-page spinner while `loading || profileLoading`.

### UI inventory
- Buttons/tabs: `Tabs` (Bubbles / Friends) switching the left-hand chat list; per-bubble and per-friend list items (click to select); message composer Send button; image-upload icon button opening an inline modal; `VoiceRecorder` (bubble chat) / `VoiceNoteRecorder` (friend chat) mic control; message-type toggle (Text/Video/Voice) and "Disappearing" `Switch` in friend chat; "Call" button in friend chat header (navigates to `/calls` with the friend pre-selected — does **not** itself start the call); "Meetups" sheet trigger + "Create New Meetup" button (bubble chat only).
- Inputs/forms: text message `Input` (both chat types), image file input (via `ImageUpload`), voice recording controls.
- Modals/dialogs: inline "Share an Image" overlay (custom `fixed inset-0` div, not a Radix `Dialog`) in both `ChatWindow` and `FriendChatWindow`; `Sheet` for bubble Meetups list; `MeetupDialog` for creating a meetup.
- Distinct states: full-page loading spinner, empty "Join some bubbles..." / "Add friends to start private messaging!" list states, empty "Select a bubble/friend to message" chat-area placeholder, populated message thread, friend-typing indicator (bouncing dots), voice-message bubble, image bubble, snap bubble.
- Workflows: land on `/messages` (optionally via `location.state.selectedFriendId` from Friends/elsewhere, or `initialFriendId` prop) → auto-selects that friend and switches to the Friends tab, clearing router state afterward → select a bubble or friend → `ChatWindow`/`FriendChatWindow` mounts, fetches last 50 messages + subscribes to a per-conversation realtime channel → type & send, or record voice, or attach image → optimistic message appended (friend chat only) → realtime INSERT event replaces/dedupes it.

### Acceptance criteria
1. Given `location.state.selectedFriendId` (e.g. arriving from Friends "Message" button) and the friends list has loaded, then that friend is auto-selected, the "Friends" tab is activated, and the router state is cleared via `history.replaceState` so a refresh doesn't re-trigger it.
2. Given a bubble is selected, when a message is sent, then it's inserted with `bubble_id` set and appears in the thread once the realtime `INSERT` fires (no optimistic append in `ChatWindow`, only in `FriendChatWindow`).
3. Given a friend chat, when a message is sent, then it appears immediately as an optimistic bubble, is inserted with a real ID, and the optimistic entry is reconciled to that ID once the insert resolves — so the later realtime echo (filtered to `recipient_id=eq.<me>`, sender = friend) does not duplicate it (dedupe checks `sender_id === friend.id`, so the sender's own optimistic message is never touched by the listener, only the reconciliation code paths update it).
4. Given a friend message insert fails, then the optimistic message is removed, an error toast is shown, and (for text) the input is repopulated with the original text so the user doesn't retype.
5. Given the friend is actively typing, when they type in their own window, a broadcast `typing` event (throttled to at most one per 1.5s) is sent on a deterministic `typing-<sortedIds>` channel; the recipient shows a "is typing…" indicator that auto-clears after 3s of inactivity.
6. Given the "Disappearing" switch is enabled and a message sent, then `is_disappearing: true` is persisted on the row — but (see risks) nothing in the fetch/render path treats it specially.
7. Given the message-type toggle is set to "Video", then submitting shows a "Video recording... coming soon" toast and no message is sent (feature stubbed, not wired to any recorder).
8. Given an image is uploaded via the modal, then a message with the format `🖼️ Image: <url>` is inserted and rendered as an inline `<img>` in the thread; content is pattern-matched by string prefix, not a dedicated message-type column, for rendering choice.
9. Given a voice note is recorded and uploaded successfully, then a message `🎵 Voice Message: <url>` is sent, a "Voice note sent!" toast appears, and the composer resets to text mode; given the upload fails, an error toast appears and no message is sent.
10. Given the chat list preview shows the last message and it starts with `🎵 Voice Message:`, then the preview text is normalized to "🎤 Voice Message" instead of the raw URL string.

### Risk-based edge cases
- [High] Disappearing messages are cosmetic only: `is_disappearing` is written on insert (`FriendChatWindow.tsx` line ~222) but no code anywhere (fetch query, render logic, a cron/edge function, or client-side timer) ever hides, expires, or deletes a message based on it — messages marked "disappearing" persist and remain visible forever, identical to normal messages. This is a functionality gap that should be verified against user-facing marketing/expectations before real-user testing.
- [High] Realtime desync / dedup fragility in `ChatWindow` (bubble chat): the `INSERT` listener always appends the new row from the broadcast payload without checking whether the current user is the sender, and there's no optimistic send — so on a slow network the sender can watch their own message "appear late," and if the realtime channel silently drops (see `console.log('ChatWindow: Subscription status:', status)` — status is only logged, never surfaced to the UI or retried), sent messages after that point never appear until a full remount/reselect of the bubble.
- [High] Message send failure has no retry affordance in `ChatWindow` (bubble chat) — the toast just reports the error; the typed text is already cleared optimistically-in-spirit (`setNewMessage('')` happens only on success in this file... actually confirm: text is cleared after the insert succeeds) — but there is no optimistic UI at all here, so a failed send simply produces an error toast without restoring anything to retry, unlike the friend-chat path which repopulates the input.
- [Med] Two independent, differently-keyed realtime channels exist per open friend conversation (`friend-messages-<friendId>-<timestamp>` for messages and `typing-<sortedIds>` for typing) plus one more per bubble chat — switching rapidly between many friends/bubbles (e.g., tab-cycling) creates and tears down several `supabase.channel` subscriptions in quick succession; if `removeChannel` cleanup races a still-connecting `subscribe()`, orphaned channels/log noise or missed initial messages are plausible.
- [Med] The "Call" button in `FriendChatWindow` only navigates to `/calls` with `state.selectedFriendId` — it does not itself invoke `startCall`, so the user must locate and click Audio/Video again on the Calls page; if they navigate away before doing so, the intent is silently lost (no toast, no auto-dial).
- [Med] Voice-note upload failure (`VoiceNoteRecorder`/`useVoiceNoteStore.uploadAndAddVoiceNote`) discards the recorded blob entirely on error with only a generic toast — there's no local retry/resend of the same recording, forcing a full re-record.
- [Low] Image upload modal is a hand-rolled `fixed inset-0` overlay rather than the shared `Dialog` primitive — it won't respond to Escape-key-close or focus-trap conventions used elsewhere in the app, an accessibility/consistency gap.
- [Low] `friends` list "Online presence dot" in the friend-list sidebar is hard-coded green (`bg-green-500`) for every friend regardless of actual presence — it is not wired to any presence channel, so it always claims everyone is online.

## Route: /calls — Calls
**Auth requirement:** logged-in. `Calls.tsx` redirects to `/auth` when `!user && !loading`, and shows a full-page spinner while `loading || pageLoading`.

### UI inventory
- Tabs: Friends (call-a-friend list with Audio/Video buttons), Bubbles (group-call list with Audio/Video buttons per bubble), History (call log).
- Buttons: per-friend "Audio"/"Video" (`startCall(friend.id, type, false)`), per-bubble "Audio"/"Video" (`startCall(bubble.id, type, true)`), History "Export" (client-side CSV download of `callHistory`, disabled when empty), `MissedCallLogDrawer` trigger (badge shows unread missed-call count), each history row opens `CallDetailDialog`.
- Modals/dialogs: `CallDetailDialog` (per-call detail: status, type, participants, duration, timestamp, "missed"/"declined" reason banner); `MissedCallLogDrawer` (`Sheet`) listing unread/read missed-call notifications with per-item "Mark seen" and "Call back."
- Distinct states: page loading spinner; empty "No friends yet" (with "Find Friends" CTA), empty "No bubbles joined" (with "Find Bubbles" CTA), empty "No call history yet"; populated history list with per-row status badges (`ended`/`missed`/`declined`/other via `Badge` variant); a `MissedCallBanner` component is imported and conditionally rendered from `missedCall` state, but **nothing in this file ever calls `setMissedCall(...)` with real data** — only `null` is ever set (see risk below), so this banner is effectively dead code today.
- Workflows: pick a friend/bubble → tap Audio/Video → `CallContext.startCall` inserts a `call_logs` row (`status: 'ringing'`), plays the outgoing ring, and renders the global `VideoCall`/`GroupVideoCall` overlay (mounted by `CallProvider`, not by this page) → for 1:1 calls a client-side timeout (default 30s, configurable via `localStorage['call-timeout-seconds']`) auto-marks the call `missed` and creates a `missed_call` notification if the callee never answers → History tab reflects the outcome once the call ends; "Call back" from the drawer re-invokes `startCall` with the stored target.

### Acceptance criteria
1. Given the Friends tab has no friends, then "No friends yet" is shown with a button routing to `/friends`.
2. Given a friend's Audio/Video button is tapped, then `startCall` fires, an outgoing ringtone starts synchronously (before any network await, to survive browser autoplay policies), and a "📞 Calling..." toast appears.
3. Given the callee never answers within the configured timeout, then the `call_logs` row transitions to `status: 'missed'`, a `missed_call` notification is written for the callee, and the caller's local call UI is cleared (`activeCall` set to null) — but note this happens on the caller's client only; if the caller's tab is closed before the timeout fires, the call log can remain `ringing` indefinitely (see risk).
4. Given a call is in the History tab with `status: 'ended'`, then duration is shown as `M:SS`; given `status` is `'missed'` or `'declined'`, a destructive-tinted phone icon is shown instead of the outgoing/incoming icon.
5. Given "Export" is clicked with a non-empty history, then a `call-history.csv` file downloads containing Date, Name, Type, Status, Duration(s) for every loaded row (capped at the last 20 calls fetched).
6. Given the Missed Calls drawer is opened, then it fetches up to 20 `missed_call` notifications with caller/bubble names hydrated, shows an unread-count badge on the trigger, and "Mark seen" / bulk actions update `notifications.read` and are reflected without a full page reload.
7. Given "Call back" is used from either the drawer or `MissedCallLogDrawer`'s callback, then `startCall` is invoked with the correct target type (`bubbleId` takes priority over `friendId` when both exist on the notification payload).
8. Given a bubble Audio/Video button is tapped, then `startCall(bubble.id, type, true)` is used and the resulting call renders `GroupVideoCall` with mesh WebRTC to every other present member.

### Risk-based edge cases
- [High] The in-page `MissedCallBanner` is unreachable dead code: `setMissedCall` is never called with a populated `MissedCallBannerData` anywhere in `Calls.tsx` (confirmed via full-file review and code search) — a user sitting on the Calls page while a call they placed goes unanswered, or while they miss an incoming call, gets **no** live in-page banner; missed-call awareness only comes from the separate `/missed-calls` route, the `MissedCallLogDrawer`, or the app-wide toast in `useRealtimeNotifications`/notification bell. This looks like an intended feature that was never wired up — worth flagging before user testing since it may be assumed working from the code's presence.
- [High] Caller-side-only timeout: the 30s (default) no-answer timeout lives entirely in `CallContext.startCall`'s `setTimeout`, guarded by a realtime channel watching for a status change. If the caller closes the tab, loses network, or navigates away before the timeout fires, the `call_logs` row is left in `status: 'ringing'` forever — it will never become `missed`, so it won't show in History or the missed-calls list, and (worse) `IncomingCallNotification`'s polling fallback on the callee's side treats any `ringing` row from the last 90s as a live incoming call, meaning a callee could still see/answer a "ringing" call whose caller already left.
- [High] Race between simultaneous accept/hangup: if the callee accepts (`call_logs.status → 'connected'`) at nearly the same instant the caller cancels (`CallContext.endCall` fetches the row, sees `status === 'ringing'` was already flipped, and no-ops the update) — but `endCall` unconditionally calls `setActiveCall(null)` on the caller's side regardless of what it read, so the caller's UI closes immediately while the callee's `VideoCall` instance believes it's connecting/connected with a peer who already tore down their `RTCPeerConnection` and media tracks. The callee is left in a "Connecting…" spinner with no further status transition until they manually hang up (the `videocall-status-<callLogId>` watcher only reacts to `declined`/`missed`/`ended`, and the caller's abandoned call may never receive one of those writes in this race).
- [Med] Two friends both start bubble/direct calls to the same person in quick succession before UI feedback settles: since `startCall` immediately sets `activeCall` (only one `activeCall` is tracked in `CallContext`, no queue), starting a second call while one is already active is not guarded against on the caller's own client (nothing disables the Audio/Video buttons while `activeCall` is already set), so a user could stomp their own in-progress call.
- [Med] Duplicate concurrent incoming calls to the same user: `IncomingCallNotification` keeps only a single `incomingCall` in state, so if two different friends both call the same recipient close together, the second `INSERT`/poll result silently replaces the first in state — the first caller's call is never explicitly declined and will only resolve via its own timeout, while the recipient never sees that a first call existed.
- [Med] `getCallTimeoutSeconds()` reads `localStorage` and is not otherwise validated against a reasonable range from the UI (no visible settings control was found for it in the pages under review) — a corrupted/edited `localStorage` value only falls back to 30s if non-finite or ≤ 0, but e.g. `"0.5"` seconds would be accepted, causing near-instant false "missed" markings.
- [Low] CSV export is capped at the last 20 fetched call logs (the `call_logs` query uses `.limit(20)`) with no "load more"/pagination, so "Export" silently produces an incomplete history for active users without any indication that more history exists.
- [Low] WebRTC/mic-camera permission failures during `startCall` from this page are only surfaced deep inside `VideoCall.initializeCall` (a global overlay mounted elsewhere) — from the Calls page itself there is no camera/mic pre-check before dialing, so a user without device permissions only discovers the failure after already committing to "Calling…" state and hearing the outgoing ring.

## Route: /missed-calls — MissedCalls
**Auth requirement:** logged-in. `MissedCalls.tsx` redirects to `/auth` when `!user && !loading` (no separate page-loading spinner state beyond the query's `isLoading`).

### UI inventory
- Buttons: "Mark all seen (N)" bulk action (only rendered when `unreadCount > 0`), per-row "Seen" (only for unread rows), per-row "Call back".
- Inputs/forms: search box (`filters by caller/bubble name, client-side`), `Tabs` filter (All / Today / This week).
- Modals/dialogs: none — flat list page.
- Distinct states: loading spinner (`react-query` `isLoading`), empty "No missed calls found." (after search/filter or when the underlying table is empty), populated list with unread rows visually highlighted (`bg-primary/5`).
- Workflows: page loads notifications of `type: 'missed_call'` (up to 100, react-query cached under key `['missed-calls', user.id]`) → derives caller/bubble profile lookups as two dependent queries keyed by the joined id list → user searches/filters client-side over the already-fetched 100 rows → "Call back" marks the notification read then **navigates to `/calls` via `window.location.href` (full page reload)** rather than actually starting the call itself.

### Acceptance criteria
1. Given a user has 0 `missed_call` notifications, then "No missed calls found." is shown regardless of the selected filter/search.
2. Given the "Today"/"This week" tab is selected, then only notifications whose `created_at` falls in that window (via `date-fns` `isToday`/`isThisWeek`) are shown; "All" shows everything fetched (capped at 100).
3. Given a search term is typed, then rows are filtered client-side by the resolved bubble name or caller's first name (case-insensitive substring); this runs entirely against already-loaded data (no server round-trip per keystroke).
4. Given "Mark all seen" is clicked with N unread visible-after-filter rows, then all N are updated to `read: true` in a single batched `.update().in('id', ids)` call, the `missed-calls` query is invalidated/refetched, and a confirmation toast is shown.
5. Given a single unmarked row's "Seen" button is clicked, then only that row updates to read and the query is invalidated.
6. Given "Call back" is clicked, then the notification is marked read and the browser performs a full navigation to `/calls` (`window.location.href = '/calls'`) — the call itself is not initiated from this page; the `callerId`/`bubbleId`/`callType` captured in `handleCallBack`'s parameters are effectively discarded (see risk).
7. Given `data.bubbleId` and `data.callerId` are both absent on a notification (malformed/legacy row), then the row still renders with "Unknown" as the title and a Call back button that is a no-op (`if (!target) return;`).

### Risk-based edge cases
- [High] "Call back" is functionally broken as an auto-dial shortcut: `handleCallBack(callerId, bubbleId, callType)` computes `target` but only ever does `window.location.href = '/calls'` — it never passes the target/type anywhere (no query string, no router state, no storage), so after the full-page reload the user lands on a generic Calls page and must manually find the same person/bubble and tap Audio or Video again. This silently defeats the entire purpose of the "Call back" button and should be confirmed as a bug, not a stub-in-progress, before user testing.
- [High] Full-page reload via `window.location.href` (rather than SPA `navigate()`) throws away all in-memory state (any other unsaved page state, the `CallContext` no-op since no call was started anyway) and is jarring UX/perf compared to every other cross-page action in this slice, which uses `react-router` navigation.
- [Med] The two dependent profile/bubble queries key off `rawItems.map(i => i.data?.callerId).join()` / `bubbleId).join()` as the react-query `queryKey` — if `rawItems` order is unstable between renders (it shouldn't be, since it's server-sorted, but any client-side re-sort would) this key would thrash and cause redundant refetches; more concretely, a malformed `data` payload (`callerId`/`bubbleId` as non-string, e.g. `null` surviving `.filter(Boolean)` incorrectly) could throw during the `[...new Set(...)]` reduction.
- [Med] No pagination beyond the first 100 notifications — a very active user (many missed group-bubble calls) will have older missed calls simply invisible on this page with no indication more exist, and no "load more."
- [Low] Search matches only the resolved caller/bubble *name*, not call type or date — searching "video" or a date string yields nothing, which may surprise QA testers expecting a more general text filter.
- [Low] Bulk "Mark all seen" operates on `filtered` (the currently visible, filtered/searched subset) not literally "all" missed calls — if a user has unread missed calls outside the current filter (e.g., they're on the "Today" tab), the button's parenthetical count and behavior only cover what's visible, which could be misread as "mark literally everything as seen."

## Route: /camera — Camera
**Auth requirement:** logged-in. `Camera.tsx` (`CameraPage`) redirects to `/auth` when `!user && !loading`.

### UI inventory
- Buttons: close/back (`X`, calls `navigate(-1)`), flash toggle (`Zap`/`ZapOff` — cosmetic, see risk), camera flip (`SwitchCamera`, toggles `facingMode` user/environment), shutter (captures a still frame from the live `<video>` to canvas → data URL), post-capture: discard/back (`X`), download (`Download`, saves the JPEG locally), "Story" quick-send, "Send To..." (opens bottom sheet), in the sheet: "My Story" toggle tile, per-friend selectable tiles (multi-select), final "Send (N)" button (disabled until at least one friend or "My Story" is selected).
- Inputs/forms: none text-based — the sheet's selections are the only "form" state (a `Set<string>` of selected friend ids + a `sendToStory` boolean).
- Modals/dialogs: bottom "Send To" sheet (custom `absolute` panel, not a Radix `Sheet`), full-screen camera UI itself is effectively a takeover view (`fixed inset-0 z-50`).
- Distinct states: live camera view (or camera-error console log with no on-screen fallback UI if `getUserMedia` throws — see risk), captured/review view, sending spinner state on the Send button (`sending`), sheet open/closed.
- Workflows: `/camera` mounts → `getUserMedia` requested with `facingMode`/1080×1920 ideal constraints → shutter captures a frame → user chooses recipients (friends and/or "My Story") → "Send" uploads the JPEG blob once to Supabase Storage (`stories` bucket) and reuses the single resulting public URL for (a) one `messages` insert per selected friend (`content: '📸 Snap: <url>'`, `message_type: 'snap'`) and (b) one `location_stories` insert if "My Story" was chosen (using best-effort live geolocation, else `0,0`) → on completion, `CameraPage.handleCapture` increments the snap score and navigates to `/messages`.

### Acceptance criteria
1. Given the page loads and camera permission is already granted, then the live preview renders using the currently selected `facingMode` (`environment` by default).
2. Given the shutter button is tapped, then the current video frame is drawn to an off-screen canvas and shown as a full-screen preview (`capturedImage` state), replacing the live camera view.
3. Given the review view is showing, then "Discard" (X) returns to the live camera and re-shows the video (note: `startCamera` is only re-triggered by a `facingMode` change effect — see risk), and "Download" saves the JPEG as `snap-<timestamp>.jpg` without sending anything.
4. Given "Send To..." is tapped, then the bottom sheet lists "My Story" plus every friend from the friendships table, each toggle independently, and the Send button is disabled until at least one is selected.
5. Given Send is tapped with ≥1 friend selected, then the image is uploaded exactly once to the `stories` storage bucket, and one `messages` row per selected friend is created referencing that same URL.
6. Given "My Story" is selected (with or without friends), then a `location_stories` row is created with a 24-hour `expires_at`, using the device's current geolocation if obtainable within 5s, else `(0, 0)`.
7. Given the send completes (success or failure), then the sheet closes, selections are cleared, and `sending` reverts to false; on failure, a destructive toast "Failed to send snap" is shown and the captured image reverts to `null` — meaning the captured photo is lost and must be retaken even though only the network step failed (see risk).
8. Given the shutter's quick "Story" button (bypassing the sheet) is tapped, then `sendToStory` is forced true and the sheet still opens (it does not immediately send) — the user must still tap the sheet's own Send button.

### Risk-based edge cases
- [High] Camera/mic permission denial has no visible recovery UI: `startCamera`'s catch block only does `console.error('Camera error:', err)` — there is no toast, no "permission denied, please enable camera access" message, and no retry button. A user who denies camera access sees a black/empty video area with a shutter button that will silently no-op (drawing an empty frame) or throw, with no guidance to fix it.
- [High] On send failure, the captured photo is discarded (`setCapturedImage(null)` runs in the `finally` block unconditionally) even though the failure could be purely transient network/storage — the user loses their capture and must retake the photo entirely rather than being offered "Retry" with the same image data still in memory.
- [Med] `facingMode` toggle re-runs `startCamera`, which stops the previous `stream` from the *stale closure* over `stream` state captured at effect-definition time (the effect's cleanup uses the `stream` value from that render, and `startCamera` itself also reads `stream` via closure) — rapid repeated camera-flip taps can leave a previous `MediaStreamTrack` running (camera light staying on / resource not released) if the timing of state updates and effect re-runs overlaps; worth a focused device test (`SwitchCamera` spammed quickly).
- [Med] Geolocation for "My Story" from the Camera flow silently falls back to `(0, 0)` (`null island`) on timeout/denial with no user-facing warning — a story could be posted with wildly wrong coordinates, and since Stories.tsx filters nearby stories by distance from `(0,0)`, such a story could appear "nearby" to users physically near the equator/prime-meridian and never appear to the actual poster's real neighbors.
- [Med] No client-side image size/type validation before the Storage upload — an extremely large canvas export (e.g., a high-resolution device camera) could produce a slow upload with only a generic "Sending..." label and no progress indicator, and no explicit timeout/cancel affordance.
- [Low] The quick "Story" button and the "Send To..." button both funnel into the same sheet (quick "Story" just pre-toggles the "My Story" tile) — there is no true one-tap "send to story immediately" path, which may not match the "quick send" affordance the two-button layout visually implies.
- [Low] `CameraPage.handleCapture` (the `onCapture` prop) fires only after a successful send inside `CameraScreen.handleSend`, per its own comment — if a caller mentally models `onCapture` as firing at shutter-press time (matching its name), the actual timing (post-send) could confuse future maintainers modifying snap-score or analytics logic.

## Route: /stories — Stories
**Auth requirement:** No route-level guard and no in-page redirect exists (`Stories.tsx` never checks `user` to redirect) — the page is effectively public. Viewing "Nearby Stories" (`location_stories`, filtered only by expiry and, if location is available, distance) requires no auth at all; only reactions, view-tracking, and "friend story ring" data silently no-op for a logged-out visitor (`if (!user) return` guards inside `handleReaction`, `fetchViews`, `fetchFriendStoryCreators`).

### UI inventory
- Buttons: "Your Story" add tile (opens `CreateStoryDialog`), "Share Story" button (also opens the dialog), per-story reaction buttons (👍 like / 💖 love / 😂 laugh — toggle on/off, mutually exclusive per user since `userReactions` stores one `reaction_type` per story per user via `upsert`), a non-interactive 👀 views button (view count only, no click handler wired despite `Button` styling), per-friend `StoryRing` avatars (horizontal scroll row) — clicking a `StoryRing` currently has no `onClick` wired from `Stories.tsx` (the component accepts an `onClick` prop but none is passed), so friend rings are decorative only on this page.
- Inputs/forms: `CreateStoryDialog`'s `Textarea` (caption/description), optional image file `Input` with live preview + remove button, duration `<select>` (1/6/12/24/48 hours).
- Modals/dialogs: `CreateStoryDialog` (Radix `Dialog`) — only submits when `userLocation` is available (button is disabled without it and without any text/image).
- Distinct states: `PageSkeleton` while `loading`; "No Stories Nearby" `EmptyState` with a "Create Your First Story" CTA; populated grid of story cards, each visually dimmed (`opacity-60`) once `expires_at` has passed (**but not removed** — see risk); story-ring row hidden entirely when no friends have active stories.
- Workflows: page loads → fetch profile, fetch nearby `location_stories` (client-side expiry + distance filtering), fetch friend story-creator avatars for the ring row, then (once stories are loaded) fetch reactions and views for the visible set, recording a view (`story_views` insert) for every not-yet-viewed story by the current user in a loop → user reacts/comments/creates a new story → `CreateStoryDialog` closes and calls `fetchStories()` to refresh the grid.

### Acceptance criteria
1. Given a story's `expires_at` is in the past, then it is excluded from the initial `fetchStories()` result entirely (filtered before being added to state) — it should never appear in the grid at all, not merely dimmed.
2. Given a story is close to expiring, then the `Badge` shows a live-computed "Xh Ym left" (or "Expired" if `ms <= 0`) — but this is computed only from data already in state and is **not re-rendered on a timer**, so "5m left" will keep displaying "5m left" indefinitely until the next full data refetch, even well past actual expiry (see risk).
3. Given "Create Your First Story"/"Share Story"/"Your Story" is used and the user grants location, then `CreateStoryDialog` allows submission once either text or an image is present; without `userLocation`, the submit button stays disabled and an error ("Location or user missing") shows if submission is forced.
4. Given a story is created successfully, then the dialog resets its local state (text/image/preview), closes, and the parent calls `fetchStories()` so the new story appears (assuming it passes the distance filter for the *current* viewer).
5. Given a reaction button (👍/💖/😂) is clicked and the user has no existing reaction on that story, then a new `story_reactions` row is upserted and the button highlights.
6. Given the same reaction type is clicked again, then the reaction is deleted (toggle-off) and the count decrements locally without a full data refetch (reactions on other stories are unaffected).
7. Given a different reaction type is clicked while one is already active for that story/user, then the old reaction is replaced (single reaction per user per story is enforced via `upsert` keyed on `(story_id, user_id)`), though the UI count-per-type is only correct after `fetchReactions()` completes (brief stale state possible).
8. Given a story is viewed for the first time by the current user, then a `story_views` row is recorded once (subsequent visits don't re-record, guarded by checking `viewedStories` before insert) and the view count refreshes.
9. Given the friend story-ring row is rendered, then the ring gradient border only appears for friends with an unexpired story (`hasUnwatched: true` — actually this flag is hardcoded `true` for every friend with *any* unexpired story regardless of whether the current user already viewed it — see risk), and "Your Story" tile always opens the create dialog regardless of ring state.

### Risk-based edge cases
- [High] Story-expiry-mid-view race: a story's expiry is only evaluated once, at `fetchStories()` time, against `new Date()`. There is no interval/timer re-checking expiry client-side, so a story that expires while the page is open remains fully visible, reactable, and commentable (only its badge text becomes stale/wrong, still showing e.g. "12m left" long after expiring) until the user manually refreshes or triggers a refetch (creating a new story, etc.). Reactions/comments/views on an already-expired-but-still-displayed story will still be written to the DB with no server-side check preventing it (no RLS/expiry enforcement was found in the client code reviewed).
- [High] `hasUnwatched` on `StoryRing` is unconditionally `true` for every friend who has at least one currently-unexpired story (`fetchFriendStoryCreators` maps every result to `hasUnwatched: true`) — it does not check the `story_views` table for whether the current user has actually viewed that friend's story yet. This means the "new story" gradient ring never turns off/plain even after the viewer has seen every story from that friend, misrepresenting unread state.
- [High] Friend `StoryRing` avatars in `Stories.tsx` have no click handler wired (the component supports `onClick` but the page doesn't pass one) — tapping a friend's story ring does nothing, which will read as a broken/dead control to a real user expecting Snapchat/Instagram-style story-ring navigation into that friend's story.
- [Med] Distance filtering for "Nearby Stories" is done entirely client-side over a `.limit(50)` fetch of *all* stories (not geographically pre-filtered server-side) — once story volume grows, relevant nearby stories can be pushed out of the top-50 by unrelated far-away stories before the distance filter ever runs, silently hiding genuinely nearby content.
- [Med] View recording loop (`fetchViews`) issues one `story_views` insert per unviewed story sequentially in a `for` loop, then performs a *second* full re-fetch of all view rows for all visible stories afterward if any insert succeeded — on a grid with many unviewed stories this is an N-insert-then-refetch pattern that is both slow and racy if the user navigates away mid-loop (partial view-recording, and the surrounding component has no cancellation on unmount).
- [Med] `CreateStoryDialog` requires `userLocation` (from `useLocation()`) to submit at all — a user who denies/hasn't granted location permission cannot post a story, with no fallback (e.g., letting them post without geo-tagging, or picking a location manually); the only feedback is a disabled button plus a generic "Location or user missing" error if forced.
- [Low] The 👀 view-count control is rendered as a `Button` (hover/focus styles, tab-stop) but has no `onClick` — a keyboard/screen-reader user will tab into an interactive-looking element that does nothing.
- [Low] `fetchProfile` in `Stories.tsx` uses `.single()` (not `.maybeSingle()`) — if a user's profile row doesn't exist yet (e.g., mid-onboarding edge case), this throws and is only caught by a generic `console.error`, leaving `profile` as `null` with no user-facing message, though this does not block viewing public stories.
- [Low] Reaction emoji counts are recomputed via `.filter()` over the full `storyReactions[story.id]` array on every render for every story card — a minor perf smell on a large grid with heavy reaction counts, not currently a functional bug.
